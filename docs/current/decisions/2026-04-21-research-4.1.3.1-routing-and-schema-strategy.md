# 2026-04-21: Iteration 4.1.3.1 Routing and Schema Strategy

## Decision

Iteration 4.1.3.1 adopts the following planning decisions:

1. `jd_facts`
- provider: Codex
- model: `gpt-5.2`
- escalation: none
- fallback: none

2. `classification`
- provider: Codex
- primary model: `gpt-5.4-mini`
- escalation model: `gpt-5.2`
- fallback: none

3. `application_surface`
- provider: Codex
- model target for 4.1.3.1: `gpt-5.2`
- fallback: none
- behavior target: **always produces a useful artifact** — `resolved`, `partial`, `negative`, or `unresolved_evidenced`. Verified employer jobs portals, aggregator-only states, stale/closed signals, and ATS-family hypotheses are all first-class outputs. Evidence and intermediate conclusions are preserved in every case. Fails closed on falsehood (fabricated URLs, cross-company links, guessed ATS deep links); never fails empty on incompleteness.

4. `research_enrichment`
- provider: Codex
- benchmark candidates: the exact Codex-available GPT-5.3 variant, `gpt-5.2`, and `gpt-5.4-mini`
- fallback: none
- final model selection is benchmark-gated rather than assumed

## Schema Strategy

Iteration 4.1.3.1 uses a **richer-canonical, compact-projection** schema strategy:

- the rich evidence-bearing shape is the canonical persisted artifact (company signals as rich objects; role summary/why-now/company-context with status, confidence, evidence; stakeholders with sources and confidence)
- compact scalar aliases are derived projections written alongside for readers that have not yet migrated — they are not the source of truth
- permissive ingress normalization absorbs alias/wrapper drift and coerces list/scalar/dict variation; grounded extras are preserved under named companion slots, never silently dropped
- strict persisted validation still blocks fabricated or unsafe data
- when a richer shape proves stable across live runs, the canonical schema evolves toward it rather than forcing repeated flattening

This means validation is **loose on shape and strict on truth**, and compatibility projections are a finite-lifetime migration surface rather than a long-term constraint on canonical truth.

## Why

The latest live local Codex-only run showed:

- `jd_facts` on `gpt-5.2` was materially more stable and complete
- `application_surface` discovered a useful verified employer jobs portal but lost it because the validator rejected benign shape drift
- `research_enrichment` returned grounded richer structures for company, role, and stakeholder research that were rejected because the persisted schema was thinner than the prompt contract naturally induced

The problem is therefore not simply model quality.
It is contract mismatch between prompts, ingress payloads, and persisted canonical models.

## Cost Consideration

Verified pricing indicates GPT-5.2 is materially more expensive than GPT-5.4 mini:

- input: about `2.33x`
- output: about `3.11x`

Therefore GPT-5.2 is reserved for stages where stability and correctness justify the cost:

- `jd_facts`
- `application_surface`
- possibly `research_enrichment` if benchmark data proves it is required

## Reasoning Effort

Official docs indicate GPT-5.2 Codex supports `low`, `medium`, `high`, and `xhigh` reasoning effort.
The local `codex exec --help` output does not expose a first-class CLI flag for this; it does expose `-c key=value` config overrides.

Decision:

- do not rely on reasoning-effort tuning in production until the exact working CLI config path is verified in this repo
- add a dedicated validation spike before introducing reasoning-effort-based routing
